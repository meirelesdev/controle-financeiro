# UI / UX Guidelines

## Design Philosophy

**Mobile-First, Offline-First, Sunlight-Readable.**

The app is used on smartphones, often outdoors. Every UI decision should optimize for small screens, one-thumb reachability, and high-contrast legibility under bright light.

---

## Tailwind CSS

### Use utility classes — no inline styles

```typescript
// Correct
el.className = 'bg-card rounded-xl p-4 shadow-sm';

// Forbidden
el.style.backgroundColor = '#1e2130';
```

### Custom design tokens (defined in `tailwind.config.js`)

| Token | Usage |
|-------|-------|
| `bg` | Page background |
| `bg-card` | Card / sheet surfaces |
| `bg-hover` | Hover / active states |
| `primary` | Brand accent (buttons, highlights) |
| `primary-dark` | Pressed / darker variant of primary |
| `danger` | Destructive actions, negative indicators |
| `warning` | Caution states |
| `muted` | Secondary text, labels |
| `subtle` | Tertiary text, placeholders |

**Opacity modifiers do not work with custom hex tokens.** Never write `bg-primary/50`. Use explicit hex or a new token instead.

### Money value colors — always use these, never override:

```
.text-income   →  #00C853  (positive / receitas)
.text-expense  →  #E53935  (negative / despesas)
```

Do not use `text-primary` or `text-danger` for financial values — use the semantic money classes.

### Reusable component classes (defined in `src/style.css` under `@layer components`)

Use these before creating new class patterns:
- `.card` — standard card surface
- `.btn-primary` — primary action button
- `.btn-danger` — destructive action button
- `.input` — form input field
- `.modal-sheet` — bottom sheet modal container
- `.bottom-nav` — fixed bottom navigation bar

## Responsiveness

- **Base styles target mobile** (320px+). Scale up with `sm:`, `md:` modifiers.
- Bottom navigation bar must always be reachable with one thumb.
- Touch targets: minimum 44×44px (`min-h-[44px] min-w-[44px]`).
- Avoid hover-only interactions — prefer tap/click as the primary trigger.
- Modals use `.modal-sheet` (slides from bottom on mobile, centered on desktop).

---

## PWA — Offline First

- **Never break offline use.** All data lives in IndexedDB; the app must function with no network.
- The Service Worker (`sw.js`) is generated by `vite-plugin-pwa` at build time. **Do not write or edit `sw.js` manually.**
- All built assets are precached by Workbox. Google Fonts use Cache-First.
- The Vite `base` is `/controle-financeiro/`. All asset paths, PWA manifest `start_url`, and `scope` must use this prefix.
- When adding new routes or assets, verify they are covered by the SW precache manifest (automatic via Vite build).

### Install / update flow

- The app registers the SW on load. On update, prompt the user to reload.
- Never force-reload silently — always surface the update notification.

## Accessibility — WCAG AA Minimum, AAA for Financial Values

### Contrast ratios

| Element | Minimum ratio | Target |
|---------|--------------|--------|
| Body text on `bg` | 4.5:1 (AA) | 7:1 (AAA) |
| Financial values (`text-income`, `text-expense`) | 4.5:1 (AA) | 7:1 (AAA) |
| Secondary / muted text | 3:1 (AA Large) | 4.5:1 |
| Interactive elements (buttons, links) | 3:1 for UI components | — |

> The target is outdoor sunlight readability. When in doubt, increase contrast.

### Semantic HTML

- Use `<button>` for actions, `<a>` for navigation — never `<div onclick>`.
- Form inputs must have associated `<label>` elements (use `for`/`id` pairing or wrap).
- Use `role` and `aria-label` on icon-only buttons.

```typescript
// Correct
const btn = document.createElement('button');
btn.setAttribute('aria-label', 'Excluir transação');
btn.innerHTML = '🗑️';

// Forbidden
const div = document.createElement('div');
div.onclick = () => deleteTransaction(id);
```

### Focus management in modals

- When a modal opens, move focus to the first interactive element inside it.
- When a modal closes, return focus to the element that triggered it.
- Trap focus within the modal while it is open (Tab/Shift+Tab cycle only inside).

## DOM Rendering Model

Views are rebuilt on every state change (`renderXxx` called again). This means:

- **Do not cache DOM references across renders.** Attach event listeners after each `innerHTML` assignment.
- Event delegation is acceptable for lists — attach one listener on the container, check `event.target`.
- Avoid attaching listeners inside loops when a single delegated listener suffices.

---

## Toast Notifications

- Success / error feedback uses short-lived toast messages (2–3 seconds).
- Validation errors inside modals: show toast **and** keep the modal open (return `false` from `Modal.onConfirm`).
- Never use `alert()` or `confirm()` for app feedback — use the in-app modal/toast pattern.
- Destructive confirmations (delete, clear all data, restore backup) require an explicit confirmation modal, not a browser `confirm()`.

---

## Month Navigation

- The `MonthPicker` component is shared across Dashboard, Transactions, and Cards views.
- Default to the current month on first load.
- Navigating months must not trigger full page reloads — re-render only the view content.

---

## Icons and Emoji

- Use emoji for category icons — consistent with the `CategoryDef.emoji` field.
- Fallback for unknown categories: `📌`.
- Keep emoji usage intentional and sparse outside of categories — do not decorate every UI element with emoji.
